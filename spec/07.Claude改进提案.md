# GoScript 语法规范改进提案

本文档针对 GoScript 语法规范的评审结果，按原始章节编号逐一给出改进方案。

> 标记说明：🔴 必须修复（矛盾/语义错误）· 🟡 建议改进 · 🟢 可选优化

---

## §2 基本数据类型

### 🟡 2-A：`Float` 应映射到 `float64`

**问题**：`Float` → `float32` 与 Go 生态习惯相悖。Go 标准库 `math` 包全部使用 `float64`，JSON 数字解析为 `float64`，浮点字面量默认 `float64`。用户直觉写 `let pi = 3.14`，如果推断为 `Float`（32 位），会出现精度丢失。

**改为**：

| GoScript 类型 | Go 映射 | 说明 |
|---------------|---------|------|
| `Float32` | `float32` | 32 位浮点 |
| `Float` | `float64` | 64 位浮点（默认） |
| `Float64` | `float64` | `Float` 的显式别名 |

浮点字面量（如 `3.14`）默认推断为 `Float`（即 `float64`）。

## §5.2 for 循环

### 🟡 5-A：for-in Map 遍历建议改为 `key, value` 顺序

**问题**：当前规范 Array 为 `value, index`，Map 为 `value, key`，两者都是 value 在前。对 Array 这合理（大多数场景只用 value），但 Map 场景中 key 在前是几乎所有语言的惯例（Go、Python、JavaScript、Kotlin），且与 Go 的 `for k, v := range m` 一致。

**建议**：Array 保留 `value, index`（高频场景优先），Map 改为 `key, value`：

```gos
// 原版
for value, key in myMap { }

// 改为
for key, value in myMap { }
```

如果希望数组和 Map 完全统一，则都改为第一位置为索引/键：

```gos
for index, value in items { }
for key, value in myMap { }
```

> 两种方案各有取舍，但至少 Map 应该 key 在前，因为"遍历 Map 取 key"的频率远高于"遍历 Map 取 value"。

## §6.6 函数类型 / §7.1 Lambda

### 🔴 6-A：零参数 Lambda 必须写 `{ -> }`，与代码块消歧

**问题**：当前规范中，Lambda 用 `{ 参数 -> 函数体 }` 标识，但零参数 Lambda 没有 `->` 标记，与代码块在语法上完全相同：

```gos
let f = { print("done") }   // Lambda？代码块？
go { print("done") }         // 歧义
```

`let callback () -> () = { print("done") }` 通过类型注解消歧，但无注解时解析器无法区分。

**改为**：零参数 Lambda 必须显式写 `-> `（参照 Kotlin）：

```gos
let f = { -> print("done") }           // 明确是 Lambda
let callback () -> () = { -> print("done") }

go { print("done") }                   // 明确是代码块（go 后面只接代码块）
defer { file.Close() }                 // 明确是代码块
```

在附录 C 的表格中，Lambda 行的消歧依据更新为"有 `->` 标记（包括零参数的 `{ -> ... }`）"。

## §8.3 类字段可变性

### 🟡 8-A：类字段应显式标注 `let`/`val`

**问题**：变量声明要求 `let`（可变）或 `val`（不可变），但类字段默认可变且无 `let` 关键字，语义不一致：

```gos
let x = 10      // 必须写 let
val y = 20      // 必须写 val

class User {
    name String      // 隐式可变，无 let 标记 ← 不一致
    val id Int       // 不可变
}
```

**改为**：类字段必须显式使用 `let` 或 `val`，与变量声明一致：

```gos
class User {
    let name String      // 可变（显式）
    val id Int           // 不可变（显式）
    private let age Int  // 可变 + 私有
}
```

主构造函数参数默认 `val`（不可变），需要可变时显式写 `let`：

```gos
class Person(
    val name String,     // 不可变（默认，val 可省略）
    let age Int,         // 可变（显式）
)
```

> 理由：主构造函数参数通常用于初始化后不再修改，默认不可变更安全。

## §9.3 接口组合

### 🟡 9-A：接口继承改用 `:` 代替 `->`

**问题**：`->` 在语言中承担过多含义（函数返回、Lambda 分隔、when 分支、函数类型），且在其他语境下 `->` 的语义是"产出/映射到"，而接口继承的语义是"来源于/继承自"，方向相反。

```gos
// 原版
interface ReadWriter -> Reader, Writer { }

// -> 的语义冲突：
// fun f() -> Int        意为"产出 Int"
// interface A -> B      意为"继承自 B"（方向相反）
```

**改为**：用 `:` 表示接口继承（与 Kotlin、C#、TypeScript 一致）：

```gos
interface ReadWriter: Reader, Writer {
    // 可以添加额外方法
}
```

`->` 的用途收敛为统一语义（产出/映射）：函数返回类型、Lambda 参数分隔、when 分支、函数类型。

## §10 类型定义

### 🟡 10-A：明确 `type X = Y` 在不同场景下的语义分类

**问题**："type X = Y 始终创建新类型（distinct type）"描述不准确，实际上 `type` 在多种场景下语义不同：

```gos
type UserID = Int64                    // distinct type（新类型包装）
type StringOrNumber = String | Int     // union type（联合类型定义）
type Person = Named & Aged            // intersection type（交叉类型定义）
type Point = { x Float, y Float }     // struct type（结构体定义）
```

**改为**：保持语法不变，但明确文档分类：

> `type X = Y` 定义命名类型 `X`，其底层结构由 `Y` 决定。`X` 是一个独立的命名类型，不可与其他类型隐式互换。根据 `Y` 的形式：
>
> - `Y` 为基本类型 → 新类型包装（distinct type），如 `type UserID = Int64`
> - `Y` 为 `A | B` → 联合类型，编译为 Go interface + marker method
> - `Y` 为 `A & B` → 交叉类型，编译为 Go struct 嵌入
> - `Y` 为 `{ fields }` → 结构类型，编译为 Go struct
> - `Y` 为字面量联合 → 枚举常量，如 `type Direction = "north" | "south"`

### 🟡 10-B：明确 `class` 与 `type` 结构体的使用边界

**问题**：`class Person(name String)` 和 `type Person = { name String }` 都编译为 Go struct，用户不清楚何时用哪个。

**改为**：在 §10 或 §8 添加明确的选择指引：

> - `class`：当需要方法、构造函数、init 块、嵌入（组合）等面向对象特性时使用
> - `type` 结构体：当定义纯数据结构（只有字段，无逻辑）时使用。方法通过扩展方法语法（§21）添加
> - 经验法则：优先用 `class`，仅在"无行为的数据载体"场景用 `type` 结构体

## §12.1 泛型

### 🟡 12-A：明确 `>>` 在泛型嵌套中的解析规则

**问题**：嵌套泛型 `Map<String, List<Int>>` 的结尾 `>>` 可能与右移运算符冲突。

**改为**：在 §12.1 补充规则：

> 在泛型参数列表内，`>>` 始终解析为两个连续的 `>` 闭合符号，不解析为右移运算符。如需在泛型上下文中进行右移运算，须用括号：`(a >> b)`。

## §13 集合类型

### 🟡 13-A：定义内置集合方法的来源与编译策略

**问题**：规范大量使用 `.filter()`, `.map()`, `.push()`, `.pop()`, `.sum()` 等方法，但 Go slice 没有这些方法。未说明它们的来源和编译方式。

**改为**：在 §13 添加一节"集合方法"：

> GoScript 为数组/切片提供内置方法，由编译器翻译为 Go 代码：
>
> | GoScript 方法 | 编译为 Go |
> |--------------|----------|
> | `arr.push(x)` | `arr = append(arr, x)` |
> | `arr.pop()` | `x := arr[len(arr)-1]; arr = arr[:len(arr)-1]` |
> | `arr.length` | `len(arr)` |
> | `arr.filter { ... }` | 编译器生成 for 循环 |
> | `arr.map { ... }` | 编译器生成 for 循环（Go 1.21+ 可用 `slices` 包） |
> | `arr.sum()` | 编译器生成累加循环 |
> | `arr.contains(x)` | 编译器生成 `slices.Contains` 或循环 |
>
> 注意：`.push()` 会修改原切片的变量绑定（等价于 `arr = append(arr, x)`），因此持有该切片引用的 `let` 变量必须为可变。

### 🔴 13-B：统一 Map 访问的接口，移除 comma-ok 解构

**问题**：Map 访问同时支持两种方式，语义矛盾：

```gos
// §13.2 — 返回可空类型
let v Int? = m["key"]

// §19.3 — comma-ok 解构
let (value, ok) = myMap["key"]
```

可空类型已经表达了"key 不存在"的语义（返回 `null`），comma-ok 变得冗余。

**改为**：Map 访问统一返回可空类型 `T?`：

```gos
let v = m["key"]             // 类型为 Int?，key 不存在则为 null
let v = m["key"] ?? 0        // 提供默认值
```

如需区分"key 存在但值为零值"和"key 不存在"，使用方法：

```gos
let exists = m.containsKey("key")    // 返回 Bool
```

从 §19.3 删除 `let (value, ok) = myMap["key"]` 的示例。

## §14 空值安全

### 🟡 14-A：明确 `?` 与 `[]` 后缀的优先级

**问题**：`Int[]?` 和 `Int?[]` 语义不同，但规范未定义后缀结合优先级。

**改为**：在 §14.1 或 §13.1 补充：

> `?` 绑定于紧邻的左侧类型。`[]` 同理。组合使用时从左到右结合：
>
> - `Int?` — 可空 Int
> - `Int[]` — Int 数组
> - `Int?[]` — 可空 Int 的数组（元素可空，数组本身非空）
> - `Int[]?` — 可空的 Int 数组（数组可空，元素非空）
> - `Int?[]?` — 可空的可空 Int 数组

### 🟡 14-B：明确 `!` 运算符的优先级规则

**问题**：`!` 同时有前缀（布尔 NOT）和后缀（非空断言、强制执行）用法，复合使用时有歧义。

**改为**：在 §3 运算符或 §14 补充优先级规则：

> - 后缀 `!`（非空断言 / 强制执行）优先级高于前缀 `!`（布尔 NOT）
> - `!x!` 解析为 `!(x!)`
> - 后缀 `!` 直接附着于表达式，中间不允许有空格：`name!.length`、`divide(a, b)!`
> - 前缀 `!` 与操作数之间不允许有空格：`!flag`

## §15 错误处理

### 🔴 15-A：解决 §15 与 §19.3 的多返回值矛盾

**问题**：§15.4 规定 Go 函数返回 `(T, error)` 时自动映射为可失败函数，附录 B 声称"用户不直接使用多返回值"。但 §19.3 给出了 `let (n, err) = fmt.Println("hello")` 的例子，直接解构了 `(int, error)` 返回值，与 §15 矛盾。

**改为**：建立统一规则——

> Go 函数的最后一个返回值为 `error` 类型时，GoScript 自动将其视为可失败函数。调用时：
>
> - 非 error 返回值正常接收
> - error 部分必须通过 `catch` / `try` / `!` 处理
>
> ```gos
> // Go 签名：func Println(a ...any) (n int, err error)
> // GoScript 中 error 自动映射，只接收非 error 部分：
> let n = try fmt.Println("hello")
> let n = fmt.Println("hello") catch { err -> 0 }
> let n = fmt.Println("hello")!
> ```
>
> Go 函数返回多个非 error 值时（如 `func f() (int, bool)`），使用解构接收：
>
> ```gos
> let (count, ok) = someGoFunc()
> ```
>
> Go 函数返回多值且最后一个是 error 时（如 `func f() (int, string, error)`），解构接收非 error 部分：
>
> ```gos
> let (count, name) = try someGoFunc()
> ```

§19.3 中 `let (n, err) = fmt.Println("hello")` 的示例应删除或替换。

### 🔴 15-B：定义 `throw`/`try` 在 goroutine 中的行为

**问题**：goroutine 没有返回值，`try` 的"向上传播"无处可去。`throw` 编译为 `return ..., err`，但 goroutine 的匿名函数的返回值不会被接收。

**改为**：在 §15 或 §17 补充规则：

> 在 `go { }` 块内部：
>
> - **禁止使用 `try`**（编译错误）：goroutine 没有调用者，错误无处传播
> - 必须使用 `catch` 就地处理或 `!` 强制执行
>
> ```gos
> go {
>     // 编译错误：try 不能在 goroutine 中使用
>     let data = try fetchData()
>
>     // 正确：catch 就地处理
>     let data = fetchData() catch { err ->
>         log.Error("fetch failed: $err")
>         return  // 退出 goroutine
>     }
>
>     // 正确：! 强制执行
>     let data = fetchData()!
> }
> ```

### 🟡 15-C：定义接口方法的可失败性标注

**问题**：Go 接口方法可能返回 `error`（如 `Read(p []byte) (n int, err error)`），但 GoScript 接口定义中没有表达"这个方法可能失败"的语法。

**改为**：可失败的接口方法用 `throws` 标注：

```gos
interface Reader {
    read(buf Byte[]) -> Int throws
}
```

或者，由编译器根据 Go 接口签名自动推断：

> 导入 Go 接口时，编译器自动检测方法签名中的 `error` 返回值，将对应方法标记为可失败。GoScript 自定义接口中，如果方法实现包含 `throw`，该方法自动视为可失败。

## §17.3 select

### 🔴 17-A：修复 select 语法，添加值绑定机制

**问题**：当前 select 语法会导致重复接收：

```gos
// 原版 — 语义错误：<-ch1 匹配时已接收一次，body 中又接收一次
select {
    <-ch1 -> { val v = <-ch1; process(v) }
}
```

Go 的 `case v := <-ch1:` 在匹配时同时接收并绑定值。GoScript 缺少这个绑定机制。

**改为**：在 case 模式中用变量声明绑定接收值：

```gos
select {
    val v = <-ch1 -> { process(v) }
    ch2 <- value -> { print("sent") }
    else -> { print("no communication") }
}
```

编译为 Go：

```go
select {
case v := <-ch1:
    process(v)
case ch2 <- value:
    fmt.Println("sent")
default:
    fmt.Println("no communication")
}
```

## §18.4 按需导入

### 🟢 18-A：别名使用 `as` 关键字代替空格分隔

**问题**：`import { Println print } "fmt"` 中 `Println print` 用空格分隔原名和别名，视觉上容易误读为两个独立导入项。

**改为**：用 `as` 关键字（与 Python/Kotlin/TypeScript 一致）：

```gos
// 原版
import { Println print, Sprintf sprintf } "fmt"

// 改为
import { Println as print, Sprintf as sprintf } "fmt"
```

## §19.3 Go 多返回值解构

### 🔴 19-A：与 §15-A 联动修正

见 §15-A 的修改。§19.3 中的示例应更新为：

```gos
// Go 函数返回 (T, error) — 用 try/catch/! 处理
let file = try os.Open("test.txt")

// Go 函数返回多个非 error 值 — 用解构
let (value, ok) = strconv.Atoi("123")   // 不对，Atoi 返回 (int, error)

// 正确的非 error 多返回值示例：
let (quot, rem) = divmod(10, 3)          // 假设 Go func divmod(a, b int) (int, int)
```

## §20.2 use（自动关闭）

### 🟡 20-A：定义 `Closeable` 协议

**问题**：`use` 方法要求类型可关闭，但规范没有定义哪些类型可以 `use`。

**改为**：定义内置接口：

```gos
interface Closeable {
    close()
}
```

> 任何实现了 `Closeable` 接口的类型（或 Go 中有 `Close() error` 方法的类型）都可以使用 `use`：
>
> ```gos
> file.use { f ->
>     f.readLine()
> }
> // 等价于：
> // defer { file.close() }
> ```
>
> `use` 块结束后自动调用 `close()`。如果 `close()` 是可失败方法，错误将被静默忽略（与 Go 中 `defer f.Close()` 的常见用法一致）。

## §21 扩展方法

### 🟡 21-A：补充扩展方法内 `this` 的编译规则

**问题**：扩展方法中直接访问字段（`return "Hello, $name"`），但编译为 Go 时需要一个 receiver 变量名。规范未说明编译策略。

**改为**：补充说明：

> 扩展方法编译为 Go 方法时，编译器自动生成 receiver 变量名（取类型名小写首字母）。方法体内对字段的直接访问自动加上 receiver 前缀：
>
> ```gos
> // GoScript
> fun Person.greet() -> String {
>     return "Hello, $name"
> }
>
> // 编译为 Go
> func (p Person) Greet() string {
>     return "Hello, " + p.Name
> }
> ```
>
> 使用 `this` 显式引用 receiver 也编译为同一结果：
>
> ```gos
> fun Person.greet() -> String {
>     return "Hello, ${this.name}"
> }
> ```

## §24 不支持的特性

### 🟡 24-A：扩充"不支持指针"的说明，描述编译器策略

**问题**：§24 简单写了"指针 — 编译器自动处理值/引用"，但 Go 的指针语义是基础性的（方法接收器、性能、nil 语义、标准库互操作），不可能一句话带过。

**改为**：将"不支持指针"扩展为详细的编译策略说明：

> GoScript 不暴露指针语法。编译器根据以下规则自动决定值传递或引用传递：
>
> **class 实例**：
> - 始终通过指针传递（编译为 `*StructName`）
> - 方法接收器始终为指针类型（允许方法修改字段）
> - 赋值和传参是引用语义（类似 Java/Kotlin）
>
> **type 结构体**：
> - 默认值传递（编译为 `StructName`）
> - 当标记为可空（`Point?`）时，使用指针（编译为 `*Point`）
>
> **基本类型**：
> - 始终值传递
> - 可空基本类型使用指针：`Int?` → `*int`，`String?` → `*string`
>
> **Go 互操作**：
> - 调用接受 `*T` 参数的 Go 函数时，编译器自动取地址
> - Go 函数返回 `*T` 时，GoScript 视为对应的非空类型 `T`（如果可能为 nil，视为 `T?`）

## 新增：内置函数清单

### 🟡 NEW-A：明确内置函数

**问题**：规范中大量使用 `print()`、`panic()` 等函数，但未定义它们。Go 的 `println` 是内置函数但通常不用于生产代码（应使用 `fmt.Println`）。

**改为**：在规范中添加内置函数一节：

> GoScript 提供以下内置函数（不需要 import）：
>
> | 函数 | 说明 | 编译为 Go |
> |------|------|----------|
> | `print(args...)` | 打印输出（带换行） | `fmt.Println(args...)` |
> | `panic(msg)` | 终止程序 | `panic(msg)` |
> | `Error(msg String)` | 创建错误实例 | `errors.New(msg)` |
> | `len(collection)` | 集合长度 | `len(collection)` |
> | `cap(collection)` | 切片容量 | `cap(collection)` |
> | `make(Type, args...)` | 底层构造（通常不直接使用） | `make(Type, args...)` |

## 新增：`type` 结构体字段分隔符

### 🟡 NEW-B：统一 type 结构体的字段分隔方式

**问题**：§10.1 中 type 结构体使用逗号分隔字段，§10.6 也用逗号，但 class 体中使用换行。规范未明确规则。

```gos
type Point = { x Float, y Float }          // 逗号
class User { name String \n val id Int }    // 换行
```

**改为**：明确两种写法均有效——

> 字段之间可以用逗号或换行分隔。单行定义推荐用逗号，多行定义推荐用换行：
>
> ```gos
> // 单行 — 逗号分隔
> type Point = { x Float, y Float }
>
> // 多行 — 换行分隔（逗号可选）
> type User = {
>     name String
>     age Int
>     email String?
> }
>
> // class 同理
> class Person(name String, age Int)     // 单行，逗号
> class Person(
>     name String,                       // 多行，尾随逗号可选
>     age Int,
> )
> ```
