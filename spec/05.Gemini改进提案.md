# GoScript 语法规范 - Gemini 改进提案

本提案基于《GoScript 语法规范》的初步评估，针对编译器实现和底层语义映射中存在的矛盾和不可行之处，提出了具体的修正方案。请结合原版规范进行对比。

## 1. 明确值类型与引用类型的语义划分

**关联原规范：**
- §8 类 (Class)
- §10.1 结构类型 (type = { })
- §24 不支持的特性 (指针)

**原设计问题：**
原规范规定 `class` 和结构体是同一概念，且“不支持指针，编译器自动处理值/引用”。这在 Go 的内存模型下会导致无法控制堆/栈分配，且扩展方法修改自身属性时缺乏指针接收者的显式声明，会导致意想不到的值拷贝问题。

**改进方案：**
明确区分 `class` 和 `type { }` 的内存与传递语义，赋予开发者控制权。

*   **`class` 默认引用语义**：
    *   `class` 定义的类型在编译为 Go 时，总是处理为**指针类型**（如 `*Person`）。
    *   实例化时分配在堆上（或由 Go 编译器逃逸分析决定），传递时传递引用。
    *   `class` 的方法编译为 Go 时，使用指针接收者 `func (this *Person)`，可以安全地修改内部状态。
*   **`type { }` 默认值语义**：
    *   `type X = { }` 定义的纯结构体在编译为 Go 时，处理为**值类型**（如 `Point`）。
    *   传递时发生值拷贝。适合定义如 `Point`, `Color`, `Rect` 等轻量级、不可变的数据载体。
    *   如果需要修改，应重新赋值，或者使用 `ref` 关键字（如果未来考虑引入）传递引用。
*   **修改 §24 不支持的特性**：
    将“指针 — 编译器自动处理值/引用”改为“指针 — 通过 `class` (引用传递) 和 `type` (值拷贝) 的严格区分来隐式管理，开发者无需直接操作 `*` 和 `&`。”

## 2. 更改泛型解析的消歧策略

**关联原规范：**
- §12.1 泛型函数
- 附录 A，决策 3

**原设计问题：**
依靠“标识符紧跟 `<`（无空格）”来区分泛型和小于号，对代码格式化工具（如 Prettier、gofmt）极不友好，容易被自动加上的空格破坏 AST 结构，导致语义突变。

**改进方案：**
放弃基于空格的解析 hack，选择以下两种更稳健的方案之一：

*   **方案 A（推荐，更 Go 风格）**：采用 `[T]` 作为泛型参数括号。
    ```gos
    // 定义
    fun map[T, U](items T[], transform (T) -> U) -> U[] { ... }
    // 调用
    let doubled = map[Int, Int](items) { it * 2 }
    ```
    *优点*：彻底解决 `< >` 歧义，与 Go 1.18+ 的泛型语法保持一致，解析极其简单。
*   **方案 B（Rust 风格）**：表达式层面的泛型调用使用 Turbofish `::<T>` 语法。
    ```gos
    // 定义保持不变
    fun map<T, U>(items T[], transform (T) -> U) -> U[] { ... }
    // 调用使用 ::< >
    let doubled = map::<Int, Int>(items) { it * 2 }
    ```
    *优点*：保留了 `< >` 用于类型定义的习惯，只在函数调用层面消歧义。

## 3. 明确底层类型联合类型的 Go 映射方式

**关联原规范：**
- §10.2 联合类型
- 附录 A，决策 22

**原设计问题：**
`type StringOrNumber = String | Int | Float` 规定编译为带 marker 方法的 Go interface。但在 Go 中，无法为基础类型（`string`, `int`）添加方法，因此这个编译策略**完全不可行**。

**改进方案：**
修改联合类型的编译映射策略。

*   **纯编译期联合类型（Type Erasure）**：
    在 GoScript 编译器前端进行严格的联合类型检查。编译到 Go 代码时，基础类型的联合类型**直接抹除为 `any`** (`interface{}`)。
    ```gos
    type StringOrNumber = String | Int
    fun process(val StringOrNumber) { ... }
    ```
    *编译到 Go：*
    ```go
    func process(val any) {
        // GoScript 编译器在幕后生成 type switch
        switch v := val.(type) { ... }
    }
    ```
    *说明*：这牺牲了 Go 层的强类型签名（暴露为 `any`），但这是 Go 语言体系下处理非结构体联合类型的唯一解。GoScript 编译器自身需要保证传递给此函数的参数必须是 `String` 或 `Int`。

## 4. 解决 Map 取值与多返回值解构的矛盾

**关联原规范：**
- §13.2 Map
- §19.3 Go 多返回值解构

**原设计问题：**
§13.2 规定 `m["key"]` 返回可空类型 `Int?`。但 §19.3 又允许 `let (value, ok) = myMap["key"]`。同一个表达式不能同时具有单返回值和多返回值的 AST 节点属性。

**改进方案：**
废弃 Map 的多返回值解构写法，全面拥抱“空值安全”体系。

*   **删除 §19.3 中关于 Map 解构的示例。**
*   统一规定：`m["key"]` 的结果就是可空类型 `T?`。
    ```gos
    let v Int? = m["key"]
    if v != null { ... }
    // 或者使用可选链和合并
    let v = m["key"] ?? 0
    ```
    *说明*：这让语言的类型系统更加一致，不需要为 map 的 index expression 做特殊的右值多态处理。

## 5. 解决 Go Error 互操作与解构的矛盾

**关联原规范：**
- §15.4 Go 互操作
- §19.3 Go 多返回值解构

**原设计问题：**
§15.4 规定 Go 返回 `(T, error)` 的函数在 GoScript 中被**自动视为**可失败函数（需要 `try/catch/!`）。但 §19.3 却展示了 `let (n, err) = fmt.Println("hello")` 这种把 `error` 当作普通值的解构。两者自相矛盾。

**改进方案：**
保持 §15.4 的优良设计（自动 error 映射），修正解构规范。

*   **严格执行 Error 自动转换**：所有来自 Go 的 `(..., error)` 返回值，在 GoScript 中 `error` 会被自动剥离，转为 GoScript 的抛错机制。
*   **修改 §19.3**：删除 `fmt.Println` 解构 `err` 的例子。明确规定：如果 Go 函数返回 `(T1, T2, error)`，在 GoScript 中调用它的结果只有 `(T1, T2)`，并且它是一个可失败调用。
    ```gos
    // 原 Go 函数：func do() (int, bool, error)
    // GoScript 调用：
    let (n, ok) = try do()
    ```

## 6. 明确 Lambda 内的 `return` 语义

**关联原规范：**
- §7 闭包与 Lambda

**原设计问题：**
规范未说明 Lambda 表达式中的 `return` 究竟是返回到 Lambda 之外（非局部返回），还是仅仅作为 Lambda 函数本身的返回值。这在使用高阶函数（如 `map`, `filter`）时会造成极大困惑。

**改进方案：**
在 §7 中增加“7.5 Lambda 的返回语义”。

*   **隐式返回**：Lambda 默认将最后一个表达式的值作为返回值（无需 `return`）。
*   **显式返回（局部返回）**：采用与 Swift/Go 匿名函数一致的规则。Lambda 中的 `return` 语句**仅代表退出当前最近的闭包作用域**，不进行非局部返回。
    ```gos
    items.map {
        if it < 0 { return 0 } // 只退出当前 Lambda 迭代，不退出外层函数
        return it * 2
    }
    ```

## 7. 明确 `catch` 表达式的作用域优先级

**关联原规范：**
- §15.3 调用可失败函数

**原设计问题：**
`catch` 作为后缀跟在表达式后面，在复杂表达式中容易产生优先级歧义。例如 `let x = 10 + divide(1, 0) catch { 0 } * 2`。

**改进方案：**
在 §15.3 中增加 `catch` 的结合性规则说明。

*   **规则**：`catch` 运算符具有**最高优先级**（类似后缀运算符），它仅紧密绑定到它左侧**最近的单个函数调用表达式**上。
*   如果需要捕获整个复杂表达式的错误，必须使用括号或块：
    ```gos
    // 捕获 divide
    let x = 10 + (divide(1, 0) catch { 0 }) * 2
    ```

## 8. 消除扩展方法中的属性遮蔽隐患 (Shadowing)

**关联原规范：**
- §21.1 作用域规则

**原设计问题：**
规范规定“如果与外部变量名冲突，receiver 字段优先”。这是一个反直觉且容易引发 Bug 的设计。外部变量被隐式覆盖，开发者极难察觉。

**改进方案：**
修改 §21.1 作用域解析优先级规则。

*   **规则更新**：在扩展方法内部，如果不使用 `this.` 前缀，发生名称冲突时，必须抛出编译时错误（Fail Fast）。
*   **编译期错误**：一旦出现 Receiver 字段名称与外部作用域变量同名（发生遮蔽），编译器**必须抛出错误**，要求开发者显式使用 `this.field` 来指代 Receiver 字段，或者重命名变量。
    ```gos
    let name = "global"

    fun Person.greet() -> String {
        // 编译错误：'name' 存在歧义，请使用 'this.name' 或显式引用局部/全局变量。
        return "Hello, $name"
    }
    ```
    *说明*：这是现代语言设计的最佳实践，不替开发者做危险的隐式覆盖决定，保证代码意图明确。
