# GoScript 语法规范改进提案汇总

本文档汇总 Claude、GPT、Gemini 三套改进提案，按《GoScript 语法规范》章节编号组织。每条改进标注提案来源、共识程度和建议优先级。

> **优先级说明**：P0 = 必须修复（语义矛盾/不可实现）· P1 = 建议改进 · P2 = 可选优化
>
> **共识标记**：⬛ 三方共识 · ◼ 两方共识 · ◻ 单方提出

## §2 基本数据类型

### ◼ 2-1：`Float` 应映射到 `float64`（P1）

**来源**：Claude 2-A / GPT 2.1

**原规范问题**：§2 类型表规定 `Float → float32`，但 §15.2 的 `divide` 编译示例输出 `float64`，存在矛盾。Go 生态中 `math` 包、JSON 解析、浮点字面量默认均为 `float64`。

**共识方案**：


| GoScript 类型 | Go 映射     | 说明            |
| ----------- | --------- | ------------- |
| `Float32`   | `float32` | 32 位浮点（需显式声明） |
| `Float`     | `float64` | 64 位浮点（默认）    |
| `Float64`   | `float64` | `Float` 的显式别名 |


浮点字面量（如 `3.14`）默认推断为 `Float`（即 `float64`）。

△提案结论：采用

## §5.2 for 循环

### ◼ 5-1：for-in 双变量顺序调整（P1）

**来源**：Claude 5-A / GPT 2.11

**原规范问题**：§5.2 规定数组为 `value, index`、Map 为 `value, key`，value 统一在前。但 Map 场景中 key 在前是 Go、Python、JavaScript、Kotlin 的通用惯例，与 Go 的 `for k, v := range m` 不一致。

**共识方案**：双变量时统一改为索引/键在前：

```gos
for item in items { }           // 单变量：value
for index, item in items { }    // 数组：index, value
for key, value in myMap { }     // Map：key, value
```

**§5.2 对应修改**：将"双变量时顺序为 `value, index`"改为 `index, value`（数组）/ `key, value`（Map）。附录 A 决策 16 同步更新。

△提案结论：采用，和Go保持一致

## §6.6 函数类型 / §7 闭包与 Lambda

### ⬛ 7-1：Lambda 与代码块消歧规则闭合（P0）

**来源**：Claude 6-A / GPT 2.7 / Gemini 6（部分）

**原规范问题**：零参数 Lambda 没有 `->` 标记，与代码块在语法上完全相同，解析器无法区分：

```gos
let f = { print("done") }   // Lambda？代码块？
go { print("done") }         // 歧义
```

**各方方案对比**：


| 方案     | 内容                                                               |
| ------ | ---------------------------------------------------------------- |
| Claude | 零参数 Lambda 必须写 `{ -> body }`，与 Kotlin 一致                         |
| GPT    | 显式参数 Lambda 必须有 `->`；隐式 `it` Lambda 仅允许尾随 lambda 或类型已知场景；其余视为代码块 |


**综合方案**：

1. 显式参数 Lambda：`{ x Int -> expr }`
2. 零参数 Lambda：`{ -> expr }`（必须有 `->` 标记）
3. 隐式 `it` Lambda：仅在尾随 lambda 或目标类型已知为函数类型时允许
4. 其余 `{ ... }` 视为代码块

**§7 / 附录 C 对应修改**：在附录 C 的 Lambda 行消歧依据更新为"有 `->` 标记（包括零参数的 `{ -> ... }`）"。

△提案结论：不采用，{} 默认就是lambda。例如：

```GoScript

{}                          // lambda，只是声明，没有被调用，也没有被赋值。

let f = { print("done") }   // lambda
f()

go { print("done") }        // go是关键字，后面的{}就是代码块
when x {}                   // 代码块
if expr {} else {}          // 代码块

fun my_function(callback ()->()){
    callback()
}
my_function {}              // lambda


let result = {
    10
}()     // 自调用闭包

```

### ◼ 7-2：Lambda 内 `return` 语义明确化（P1）

**来源**：GPT 2.7 / Gemini 6

**原规范问题**：§7 未说明 Lambda 中 `return` 是局部返回（退出 Lambda）还是非局部返回（退出外层函数）。

**共识方案**：Lambda 中 `return` 仅代表退出当前 Lambda 作用域，不进行非局部返回（与 Swift/Go 匿名函数一致）。建议在 §7 新增 §7.5 章节。

```gos
items.map {
    if it < 0 { return 0 }    // 只退出当前 Lambda，不退出外层函数
    return it * 2
}
```

△提案结论：采用

## §8 类

### ◻ 8-1：类字段应显式标注 `let`/`val`（P1）

**来源**：Claude 8-A

**原规范问题**：§1 要求变量声明用 `let`/`val`，但 §8.3 类字段默认可变且无 `let` 关键字，语义不一致。

**方案**：类字段必须显式使用 `let`（可变）或 `val`（不可变），与变量声明一致：

```gos
class User {
    let name String      // 可变（显式）
    val id Int           // 不可变（显式）
    private let age Int  // 可变 + 私有
}
```

△提案结论：不采用。类的成员变量应该默认可见。

```gos
class User {
    name String      // 可变（默认）
    val id Int       // 不可变（显式）
    private age Int  // 可变 + 私有
}
```

### ◻ 8-2：可见性关键字语义修正（P1）

**来源**：GPT 2.10

**原规范问题**：§8.7 中 `private` 实际含义是"包内可见"（Go 小写首字母），与其他语言中 `private = 类内部可见` 的直觉冲突。

**方案**：引入三级可见性：


| 关键字          | 语义      | Go 编译   |
| ------------ | ------- | ------- |
| `public`（默认） | 包外可见    | 首字母大写   |
| `internal`   | 包内可见    | 首字母小写   |
| `private`    | 仅类型内部可见 | 编译器静态检查 |

△提案结论：不采用，private对应的就是Go中首字母小写。

### ⬛ 8-3：明确 class 与 type 结构体的值/引用语义（P0）

**来源**：Claude 24-A / GPT 2.4、2.9 / Gemini 1

**原规范问题**：§24 简单写"不支持指针，编译器自动处理"，但 Go 的指针语义是基础性的，`class` 和 `type {}` 都编译为 struct 时值/引用不确定。

**共识方案**：


| 类型定义           | 传递语义     | Go 编译                  |
| -------------- | -------- | ---------------------- |
| `class`        | 引用语义（指针） | `*StructName`，方法用指针接收器 |
| `type X = { }` | 值语义（拷贝）  | `StructName`，适合轻量数据载体  |
| 基本类型           | 值传递      | 原生类型                   |
| 可空类型 `T?`      | 指针       | `*T`                   |


△提案结论：采用。

## §9.3 接口组合

### ◻ 9-1：接口继承改用 `:` 代替 `->`（P1）

**来源**：Claude 9-A

**原规范问题**：`->` 在语言中承担函数返回、Lambda 分隔、when 分支、函数类型等多重含义。接口继承（"来源于"）与 `->` 的产出语义方向相反。

**方案**：

```gos
// 原版
interface ReadWriter -> Reader, Writer { }

// 改为（与 Kotlin、C#、TypeScript 一致）
interface ReadWriter: Reader, Writer { }
```

`->` 用途收敛为统一的"产出/映射"语义。附录 A 决策 9 同步更新。

△提案结论：采用。

## §10 类型定义

### ◼ 10-1：明确 `type X = Y` 的语义分类（P1）

**来源**：Claude 10-A、10-B / GPT 2.9

**原规范问题**：§10 声称"type X = Y 始终创建新类型（distinct type）"，但实际在联合类型、交叉类型、结构体等场景下语义不同。`class` 和 `type {}` 的使用边界不清。

**各方方案对比**：


| 方案     | 内容                                                                                          |
| ------ | ------------------------------------------------------------------------------------------- |
| Claude | 保持语法不变，补充文档分类（基本类型包装 / 联合 / 交叉 / 结构 / 枚举常量）；添加 class vs type 选择指引                           |
| GPT    | 语法层面拆分：`type` = 类型别名、`newtype` = distinct 新类型、`class` = 引用语义对象、`record`/`type {}` = 值语义数据载体 |


**综合建议**：至少需要在文档中明确 `type X = Y` 根据 `Y` 的形式所表达的不同语义，并给出 `class` vs `type` 的选择指引。是否引入 `newtype` 关键字可进一步讨论。

△提案结论：采用 Claude 的方案（保持语法不变）。

### ⬛ 10-2：联合类型的 Go 映射策略修正（P0）

**来源**：GPT 2.3 / Gemini 3 / Claude（隐含于 10-A）

**原规范问题**：§10.2 声称 `type StringOrNumber = String | Int | Float` 编译为带 marker 方法的 Go interface，但 Go 无法为基础类型（`string`、`int`）添加方法，此编译策略不可行。

**各方方案对比**：


| 方案     | 内容                                                                       |
| ------ | ------------------------------------------------------------------------ |
| GPT    | 拆分为"约束联合"（`constraint`，仅泛型约束）和"值联合"（`union`，成员必须是命名类型）；原始类型值联合 v0.2 暂不支持 |
| Gemini | 编译期类型擦除：基础类型联合编译为 `any`（`interface{}`），由 GoScript 编译器保证类型安全              |


**综合建议**：需要明确区分泛型约束中的类型集合（可直接映射 Go type set）和运行时值联合类型（需要包装类型或类型擦除）。

△提案结论：采用 Gemini 方案。

## §12 泛型

### ⬛ 12-1：泛型语法消歧（P0）

**来源**：Claude 12-A / GPT 2.5 / Gemini 2

**原规范问题**：§12.1 依赖"标识符紧跟 `<`（无空格）"消歧泛型与比较运算符，对格式化工具极不友好，空格变化会破坏 AST。嵌套泛型 `Map<String, List<Int>>` 的 `>>` 与右移运算符冲突。

**各方方案对比**：


| 方案     | 内容                                             |
| ------ | ---------------------------------------------- |
| Claude | 保留 `<>`，补充 `>>` 在泛型上下文解析为两个 `>` 的规则            |
| GPT    | 统一改用 `[]` 泛型语法（与 Go 1.18+ 一致），彻底移除空格消歧规则       |
| Gemini | 方案 A：改用 `[]`（推荐）；方案 B：表达式层面用 Turbofish `::<T>` |


**综合建议**：GPT 和 Gemini 均推荐改用 `[]`。如保留 `<>` 则至少需补充 Claude 提出的 `>>` 解析规则。

```gos
// [] 方案示例
fun map[T, U](items T[], f (T) -> U) -> U[] { ... }
class Box[T](value T) { }
```
△提案结论：采用Claude方案（保留`<>`）

## §13 集合类型

### ◻ 13-1：定义内置集合方法的来源与编译策略（P1）

**来源**：Claude 13-A

**原规范问题**：规范大量使用 `.filter()`、`.map()`、`.push()`、`.pop()`、`.sum()` 等方法，但 Go slice 没有这些方法，未说明编译方式。

**方案**：在 §13 添加集合方法编译映射表：


| GoScript 方法          | 编译为 Go                                         |
| -------------------- | ---------------------------------------------- |
| `arr.push(x)`        | `arr = append(arr, x)`                         |
| `arr.pop()`          | `x := arr[len(arr)-1]; arr = arr[:len(arr)-1]` |
| `arr.length`         | `len(arr)`                                     |
| `arr.filter { ... }` | 编译器生成 for 循环                                   |
| `arr.map { ... }`    | 编译器生成 for 循环                                   |
| `arr.contains(x)`    | `slices.Contains` 或循环                          |

△提案结论：不采用，没有这些方法，这些方法只是为了示范代码虚构/假设出来的，编译器当成正常的标识符即可，不是内置函数，不处理，编译时遇到没有的方法，默认报错即可。

### ⬛ 13-2：统一 Map 访问接口，移除 comma-ok 解构（P0）

**来源**：Claude 13-B / GPT 2.14 / Gemini 4

**原规范问题**：§13.2 规定 `m["key"]` 返回可空类型 `Int?`，但 §19.3 允许 `let (value, ok) = myMap["key"]`。同一表达式不能同时具有单返回值和多返回值语义。

**共识方案**：Map 访问统一返回可空类型 `T?`，删除 §19.3 中 Map 的 `(value, ok)` 解构示例：

```gos
let v = m["key"]             // 类型为 Int?
let v = m["key"] ?? 0        // 提供默认值
let exists = m.containsKey("key")   // 判断 key 是否存在
```
△提案结论：采用。

## §14 空值安全

### ◻ 14-1：明确 `?` 与 `[]` 后缀的优先级（P1）

**来源**：Claude 14-A

**原规范问题**：`Int[]?` 和 `Int?[]` 语义不同，规范未定义后缀结合优先级。

**方案**：`?` 和 `[]` 绑定于紧邻的左侧类型，从左到右结合：

- `Int?` — 可空 Int
- `Int[]` — Int 数组
- `Int?[]` — 可空 Int 的数组（元素可空，数组非空）
- `Int[]?` — 可空的 Int 数组（数组可空，元素非空）

△提案结论：采用。

### ◼ 14-2：`!` 运算符语义拆分（P0）

**来源**：Claude 14-B / GPT 2.6

**原规范问题**：`!` 同时用于非空断言（§14.4 `name!`）和可失败调用强制执行（§15.3 `divide(...)!`），后缀 `!` 与前缀 `!`（布尔 NOT）复合使用时有歧义。

**各方方案对比**：


| 方案     | 内容                                                |
| ------ | ------------------------------------------------- |
| Claude | 保留双重用法，补充优先级规则：后缀 `!` 优先于前缀 `!`，`!x!` 解析为 `!(x!)` |
| GPT    | 拆分语义：`expr!` 仅用于非空断言；强制执行改为前缀 `try! expr`         |


```gos
// GPT 方案
let len = name!.length           // 非空断言（保留）
let x = try! divide(10, 2)      // 强制执行（新语法）
```

**综合建议**：GPT 的 `try!` 方案消歧更彻底，避免了同一运算符的双重语义。

△提案结论：采用GPT方案。

## §15 错误处理

### ⬛ 15-1：解决 §15 与 §19.3 的多返回值矛盾（P0）

**来源**：Claude 15-A / GPT 2.2 / Gemini 5

**原规范问题**：§15.4 规定 Go `(T, error)` 自动映射为可失败函数，附录 B 声称"用户不直接使用多返回值"。但 §19.3 给出 `let (n, err) = fmt.Println("hello")`，直接解构 `error` 返回值，与 §15 矛盾。

**共识方案**：

1. Go 函数最后一个返回值为 `error` 时，自动映射为可失败函数，`error` 部分必须通过 `catch`/`try`/`try!` 处理
2. Go 函数返回多个非 error 值时（如 `(int, bool)`），使用解构接收
3. GoScript 自定义函数不允许声明多返回值

```gos
// Go: func Open(name string) (*File, error)
let file = try os.Open("test.txt")

// Go: func Lookup(k string) (int, bool) — 无 error
let (v, ok) = lookup("a")

// Go: func ReadX() (int, bool, error)
let (n, ok) = try readX()
```

**§19.3 对应修改**：删除 `let (n, err) = fmt.Println("hello")` 和 `let (value, ok) = myMap["key"]` 示例。

△提案结论：采用。

### ◻ 15-2：定义 goroutine 中 `throw`/`try` 的行为（P0）

**来源**：Claude 15-B

**原规范问题**：goroutine 没有返回值，`try` 的"向上传播"无处可去。

**方案**：在 `go { }` 块内禁止使用 `try`（编译错误），必须用 `catch` 就地处理或 `!`/`try!` 强制执行：

```gos
go {
    // 编译错误：try 不能在 goroutine 中使用
    let data = try fetchData()

    // 正确：catch 就地处理
    let data = fetchData() catch { err ->
        log.Error("fetch failed: $err")
        return
    }
}
```

△提案结论：采用。

### ◻ 15-3：接口方法的可失败性标注（P1）

**来源**：Claude 15-C

**原规范问题**：Go 接口方法可能返回 `error`，但 GoScript 接口定义中无法表达"方法可能失败"。

**方案**：可失败的接口方法用 `throws` 标注，或由编译器根据 Go 签名自动推断：

```gos
interface Reader {
    read(buf Byte[]) -> Int throws
}
```

△提案结论：不采用。
应该改为：可失败方法返回值类型前加叹号!：
```gos
    interface Reader {
        read (buf Byte[]) -> !Int               // 可失败方法
    }
    // 实现类 — 返回类型也是 !Int
    class MyReader {
        fun read(buf Byte[]) -> !Int {
            if buf.length == 0 { throw Error("empty") }
            return buf.length
        }
    }
```

### ◼ 15-4：明确 `catch` 表达式的优先级（P1）

**来源**：GPT 2.12 / Gemini 7

**原规范问题**：`catch` 作为后缀跟在表达式后，复杂表达式中优先级不明。

**共识方案**：`catch` 作为后缀运算符，仅绑定左侧最近的函数调用表达式。复杂场景须加括号：

```gos
let x = 10 + (divide(1, 0) catch { 0 }) * 2
```

△提案结论：不采用。
应该改为：
catch 作为后缀运算符，优先级高于所有算术运算符（仅次于函数调用 () 和成员访问 .）。无需括号，catch 自然绑定左侧最近的调用表达式，右侧块 { } 为明确边界。括号为可选项。
```gos
// 等价于：10 + (divide(1, 0) catch { 0 }) * 2
let x = 10 + divide(1, 0) catch { 0 } * 2
// a()失败就返回0，然后转为字符串
let x = a()catch{0}.toString()
```

## §17.3 select

### ◼ 17-1：修复 select 语法，添加值绑定机制（P0）

**来源**：Claude 17-A / GPT 2.8

**原规范问题**：当前 `<-ch1 -> { val v = <-ch1; process(v) }` 会导致重复接收。Go 的 `case v := <-ch1:` 在匹配时同时接收并绑定值，GoScript 缺少此机制。

**各方方案对比**：


| 方案     | 语法                                                              |
| ------ | --------------------------------------------------------------- |
| Claude | `val v = <-ch1 -> { process(v) }`                               |
| GPT    | `case v <- ch1 -> { process(v) }`，引入 `case` 关键字，默认分支用 `default` |


```gos
// Claude 方案
select {
    val v = <-ch1 -> { process(v) }
    ch2 <- value -> { print("sent") }
    else -> { print("no communication") }
}

// GPT 方案
select {
    case v <- ch1 -> { process(v) }
    case ch2 <- value -> { print("sent") }
    default -> { print("no communication") }
}
```

△提案结论：采用GPT方案。

## §18.4 按需导入

import { Println as print, Sprintf as sprintf } "fmt"

## §19.3 Go 多返回值解构

### ⬛ 19-1：与 §15-1 联动修正（P0）

**来源**：Claude 19-A / GPT 2.2 / Gemini 5

## §21 扩展方法

### ⬛ 21-1：扩展方法命名冲突策略（P1）

**来源**：Claude 21-A / GPT 2.13 / Gemini 8

**原规范问题**：§21.1 规定"名称冲突时 receiver 字段优先"，这是隐式遮蔽，容易引发难以察觉的 Bug。

**共识方案（GPT + Gemini）**：名称冲突时直接**编译报错**，要求开发者显式使用 `this.field` 或重命名变量：

```gos
let name = "global"

fun Person.greet() -> String {
    // 编译错误：'name' 存在歧义
    return "Hello, $name"

    // 正确：显式使用 this
    return "Hello, ${this.name}"
}
```

**综合建议**：采用编译报错策略（Fail Fast），更符合现代语言设计最佳实践。

△提案结论：采用。
